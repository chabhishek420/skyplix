## User Constraints

### Phase Boundary
Close the highest-value Keitaro PHP backend parity gaps that remain after Phase 7, with emphasis on traffic-engine entry/dispatcher behavior and admin controller contracts that materially affect routing or operator workflows.

This phase does **not** include building the real dashboard UI. It also does **not** aim to port every peripheral Keitaro platform module one-for-one unless a direct dependency on traffic/admin parity is proven.

### Implementation Decisions
- Parity target and prioritization
  - Judge parity from the live source trees and behavior, not from existing repo docs or same-name route claims.
  - Prioritize gaps that affect real traffic handling, postback behavior, stream operations, admin workflows, and contracts the upcoming UI would rely on.
  - Use the original PHP source in `reference/Keitaro_source_php/` as the source of truth for what counts as implemented, partial, or missing.
- Dispatcher and public entry strategy
  - Treat the missing reference traffic dispatcher surfaces as first-class audit items: gateway redirect, KClientJS, KTRK, robots, and update tokens.
  - If a missing PHP dispatcher can be represented cleanly by an existing consolidated Next.js route without losing important behavior, that mapping is allowed, but it must be explicit and verifiable.
  - Keep public traffic behavior centered on the shared pipeline/runtime modules in `src/lib/tds/` instead of duplicating logic across route handlers.
- Contract fidelity for same-name resources
  - Same-name parity routes are not considered "done" if they only expose generic CRUD while the PHP controller exposes materially different actions or contract semantics.
  - Upgrade partial same-name resources before calling them parity-complete. The current known partials include reports, labels, triggers, Facebook integration, and AppsFlyer integration.
  - Generic CRUD is acceptable only when the reference controller is effectively CRUD-shaped for real operator workflows.
- Scope boundary for missing controllers
  - Keep high-value traffic/admin parity in scope for this phase.
  - Defer low-value or peripheral platform modules unless they are required by the operator workflows or the forthcoming UI.
  - Explicitly record deferred controllers so the project has an accountable "not now" list rather than vague omission.
- Codex's Discretion
  - Exact decomposition of the work into plans after research.
  - Whether some missing PHP controllers should be implemented as distinct routes or as clearly documented consolidated mappings.
  - Exact response-shape normalization where the PHP contract and current Next.js constraints differ but the operator-facing behavior can still be preserved.
- Deferred Ideas
  - Phase 8 UI work: building the real operator dashboard remains the next phase after backend parity closure.
  - Peripheral platform modules such as branding, self-update, cleaner, diagnostics-heavy tooling, and similar non-core ops areas remain deferred unless a hard dependency emerges.
  - Any product expansion beyond Keitaro parity and the real admin UI is out of scope for this phase.

## Summary

The direct source audit says Phase 7.1 should not be a broad cleanup pass. The highest-value work is still concentrated in two places: missing public dispatcher surfaces and admin endpoints whose current Next.js shape does not match the reference controller contract. The reference router exposes `gateway.php`, `robots.txt`, `_update_tokens`, `click_api/vN`, `api.php`, `Ktrk`, `KClientJS`, and the LP-to-offer flow. The current app already covers `click`, `click/json`, `lp/offer`, `postback`, and `safe`, but it does not yet expose the full reference dispatcher set, and the current `gateway-context.ts` is a general redirect helper rather than the reference JWT-decoding gateway contract.

On the admin side, the routes that look "done" by name are still partial in behavior. `reports`, `labels`, `triggers`, `integrations/facebook`, and `integrations/appsflyer` all expose useful functionality, but each one diverges from the PHP controller surface in a way that matters for parity. Phase 7.1 should therefore close dispatcher gaps first, then normalize the partial controller contracts that the UI will depend on, and explicitly defer peripheral platform modules that do not affect traffic routing or operator workflows.

**Primary recommendation:** implement the five missing traffic dispatchers first, starting with `gateway.php` and `_update_tokens`, then close the known partial admin contracts (`reports`, `labels`, `triggers`, `Facebook`, `AppsFlyer`) before touching peripheral platform modules.

## Architecture Patterns

- Public traffic routes should stay thin and either delegate to `src/lib/tds/pipeline` or to a dispatcher-specific context/helper that still uses the shared runtime.
- Admin parity routes should follow the existing `checkAuth(request)` pattern and keep request validation explicit for any mutation or contract-shaped response.
- Same-name resources must be judged by action surface, response shape, and side effects, not by route name alone.
- If the PHP controller is effectively a contract endpoint rather than CRUD, the Next.js port should expose the same contract shape, even if the path changes.
- Use the reference router in `reference/Keitaro_source_php/application/Core/Router/TrafficRouter.php` and the PHP controller files under `reference/Keitaro_source_php/application/Component/*/Controller/` as the parity baseline, not repo docs.

## Concrete Prioritization

### Implement Now
1. `gateway.php` parity first.
   - The reference gateway contract is token-driven and user-agent-sensitive.
   - The current `src/lib/tds/contexts/gateway-context.ts` is not equivalent because it performs general campaign/stream redirect selection instead of decoding the reference gateway token flow.
   - This surface is externally visible and already referenced by the current redirect logic, so it is the highest-risk gap.
2. `_update_tokens` and `robots.txt` next.
   - `UpdateTokensDispatcher` is a side-effect endpoint that should not be approximated with generic CRUD.
   - `RobotsDispatcher` is simple, easy to verify, and directly affects cloaking/indexing behavior.
3. `Ktrk` and `KClientJS`.
   - These are script-like public entry points that serialize click state for embedded integrations.
   - They should be implemented as thin wrappers around the shared traffic runtime, not as ad hoc response builders.
4. Partial admin contracts.
   - `reports` needs explicit report-definition and campaign-stat actions, not only generic analytics output.
   - `labels` in the current port is entity CRUD, but the reference `Reports/LabelsController` is a report helper contract, so it needs a separate parity surface or an explicit remap.
   - `triggers` needs stream-trigger assignment and the target/condition/action lookup endpoints, not only trigger-row CRUD.
   - `Facebook` and `AppsFlyer` need their reference description/config contract behavior, not just settings storage.

### Explicitly Defer
- `Branding`, `Cleaner`, `Diagnostics`, `SelfUpdate`, `Migrations`, `System`, `Status`, `Logs`, `Home`, `Editor`, `Trends`, `GeoDb`, and `GeoProfiles` controllers unless a hard dependency emerges.
- `Users` extras such as `ApiKeys`, `Profile`, `Preferences`, and resource helpers unless the UI phase proves it needs them.
- `CampaignIntegration` code-presets and other editor-only convenience controllers unless the upcoming UI explicitly requires them.
- `FavouriteReports`, `ExportedReports`, and similar secondary report-management surfaces unless they block a core operator workflow.
- Stream-adjacent helper controllers such as `Collections`, `FavouriteStreams`, `StreamEvents`, `StreamTypes`, and `StreamSchemas` unless a specific UI or dispatcher dependency proves otherwise.

## Common Pitfalls

- `[HIGH]` Treating route presence as parity completion. The current `reports`, `labels`, `triggers`, `facebook`, and `appsflyer` routes exist but do not fully match the PHP controller contracts.
- `[HIGH]` Reusing `gateway-context.ts` as if it were the reference gateway dispatcher. It is a useful helper, but it is not the same contract as `gateway.php`.
- `[HIGH]` Assuming the CRUD-shaped `labels` route covers `Reports/LabelsController`. The PHP controller is report-oriented, not entity CRUD.
- `[HIGH]` Assuming the CRUD-shaped `triggers` route covers `TriggersController`. The PHP controller is a stream-trigger assignment and metadata endpoint.
- `[MEDIUM]` Pulling in peripheral platform modules because they are adjacent to the admin area. Phase 7.1 should stay focused on traffic and operator-critical parity.
