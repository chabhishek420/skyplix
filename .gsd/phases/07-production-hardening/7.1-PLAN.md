# Plan 7.1: Prometheus Metrics Instrumentation

## Objective
Add Prometheus metrics to the core hot path (click pipeline, ClickHouse writer, cache) and expose a `/metrics` endpoint. This is the foundation for all monitoring — nothing else in Phase 7 works without observable data.

## Context
- .gsd/SPEC.md — Line 52: `prometheus/client_golang` specified as metrics stack
- internal/server/server.go — Server struct, dependency injection site
- internal/server/routes.go — Route definitions, health endpoint exists at `/api/v1/health`
- internal/queue/writer.go — ClickHouse batch writer, flush loop
- internal/pipeline/pipeline.go — Pipeline.Run(), stage execution loop
- internal/cache/cache.go — Entity cache with warmup
- go.mod — No prometheus dependency yet

## Tasks

<task type="auto">
  <name>Add Prometheus dependency and create metrics package</name>
  <files>
    go.mod
    internal/metrics/metrics.go
  </files>
  <action>
    1. Run `go get github.com/prometheus/client_golang/prometheus github.com/prometheus/client_golang/prometheus/promhttp github.com/prometheus/client_golang/prometheus/promauto`
    2. Create `internal/metrics/metrics.go` with pre-registered metrics using `promauto`:
       - `skyplix_http_requests_total` (CounterVec: method, path, status_code) — total HTTP requests
       - `skyplix_http_request_duration_seconds` (HistogramVec: method, path) — latency histogram with buckets [0.001, 0.0025, 0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0]
       - `skyplix_pipeline_duration_seconds` (HistogramVec: level) — click pipeline execution time (L1/L2)
       - `skyplix_pipeline_stages_duration_seconds` (HistogramVec: stage) — per-stage timing (top-level only)
       - `skyplix_clicks_total` (Counter) — total clicks processed
       - `skyplix_clicks_bot_total` (Counter) — bot clicks detected
       - `skyplix_clickhouse_flushes_total` (CounterVec: table, status) — CH flush count (success/error)
       - `skyplix_clickhouse_flush_duration_seconds` (Histogram) — CH flush latency
       - `skyplix_clickhouse_batch_size` (Histogram) — records per flush
       - `skyplix_clickhouse_channel_depth` (GaugeVec: table) — current channel buffer depth
       - `skyplix_cache_operations_total` (CounterVec: operation, status) — cache hit/miss/error
       - `skyplix_active_connections` (GaugeVec: service) — pg/valkey/ch connection count
    3. Use `promauto.NewXxx()` — NOT manual registration. This avoids duplicate registration panics.
    4. Export all as package-level vars. No init() functions.
  </action>
  <verify>go build ./internal/metrics/...</verify>
  <done>internal/metrics/metrics.go compiles with all 12 metric families defined</done>
</task>

<task type="auto">
  <name>Instrument hot path and expose /metrics endpoint</name>
  <files>
    internal/server/routes.go
    internal/server/server.go
    internal/queue/writer.go
    internal/pipeline/pipeline.go
  </files>
  <action>
    1. In `routes.go`: Add `r.Handle("/metrics", promhttp.Handler())` BEFORE the Chi middleware group (public, unauthenticated). Import `promhttp`.
    2. In `routes.go` `handleClick()` and `handleClickL2()`: After `time.Since(start)`, record:
       - `metrics.PipelineDuration.WithLabelValues("L1").Observe(elapsed.Seconds())`
       - `metrics.ClicksTotal.Inc()`
       - If `payload.RawClick.IsBot`: `metrics.ClicksBotTotal.Inc()`
    3. In `pipeline.go` `Run()`: Add timing around the entire stage loop and emit `metrics.PipelineDuration`.
    4. In `writer.go` `flushClicks()`:
       - Time the flush: `start := time.Now()`
       - On success: `metrics.ClickHouseFlushesTotal.WithLabelValues("clicks", "success").Inc()`
       - On error: `metrics.ClickHouseFlushesTotal.WithLabelValues("clicks", "error").Inc()`
       - `metrics.ClickHouseFlushDuration.Observe(time.Since(start).Seconds())`
       - `metrics.ClickHouseBatchSize.Observe(float64(len(records)))`
    5. Same for `flushConversions()` with label `"conversions"`.
    6. In `writer.go` `Run()`: Every tick, emit channel depth gauge:
       - `metrics.ClickHouseChannelDepth.WithLabelValues("clicks").Set(float64(len(w.clickChan)))`
       - `metrics.ClickHouseChannelDepth.WithLabelValues("conversions").Set(float64(len(w.convChan)))`
    7. Do NOT add middleware that wraps every request — only instrument the hot path handlers explicitly. The `/metrics` endpoint itself must NOT be instrumented (Prometheus scrapes it).
  </action>
  <verify>go build ./... && curl -s http://localhost:8105/metrics 2>/dev/null | head -5 || echo "Build passes (runtime test requires running server)"</verify>
  <done>/metrics endpoint serves Prometheus text format; pipeline and writer metrics are recorded on every click</done>
</task>
