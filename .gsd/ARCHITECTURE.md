ZAI TDS (`zai-tds`) is a production-grade, open-source Go rewrite of Keitaro TDS v9 ‚Äî a closed-source PHP Traffic Distribution System used by affiliate marketers to route, track, and optimize paid traffic. ## Click Pipeline

### Level 1 ‚Äî Campaign Click (23 stages)
Triggered when a visitor hits `/{campaign_alias}`.

```
 1.  DomainRedirect              ‚Äî Handle domain-level campaign redirects
 2.  CheckPrefetch               ‚Äî Detect and discard browser prefetch requests
 3.  BuildRawClick               ‚Äî Extract IP, UA, referrer, sub_ids, costs from request
 4.  FindCampaign                ‚Äî Lookup campaign from Valkey cache by alias/id
 5.  CheckDefaultCampaign        ‚Äî Fall back to default campaign if none found
 6.  UpdateRawClick              ‚Äî Enrich click with geo (GeoIP), device (UA parser)
 7.  CheckParamAliases           ‚Äî Resolve traffic source parameter aliases
 8.  UpdateCampaignUniqueness    ‚Äî Check/set unique visitor flag (campaign-level)
 9.  ChooseStream                ‚Äî Filter streams, pick matching by position or weight
10.  UpdateStreamUniqueness      ‚Äî Check/set unique visitor flag (stream-level)
11.  ChooseLanding               ‚Äî Rotate landing page (weighted round-robin)
12.  ChooseOffer                 ‚Äî Rotate offer (weighted round-robin)
13.  GenerateToken               ‚Äî Generate cryptographically random click token
14.  FindAffiliateNetwork        ‚Äî Resolve affiliate network postback config
15.  UpdateHitLimit              ‚Äî Check daily/total click caps (Valkey counters)
16.  UpdateCosts                 ‚Äî Apply cost model from traffic source params
17.  UpdatePayout                ‚Äî Calculate expected payout
18.  SaveUniquenessSession       ‚Äî Persist uniqueness flags to Valkey session
19.  SetCookie                   ‚Äî Write visitor_code + session cookies
20.  ExecuteAction               ‚Äî Build response (302, meta-redirect, proxy, HTML, 404)
21.  PrepareRawClickToStore      ‚Äî Serialize click data for async storage
22.  CheckSendingToAnotherCampaign ‚Äî Handle ToCampaign action recursion (max 10 hops)
23.  StoreRawClicks              ‚Äî Push to async channel ‚Üí ClickHouse batch writer
```

### Level 2 ‚Äî Landing Click (13 stages)
Triggered when visitor clicks through from landing page to offer.
The `visitor_code` cookie ties Level 1 and Level 2 together.

```
 1.  FindCampaign
 2.  UpdateParamsFromLanding     ‚Äî Merge landing-level params into click
 3.  CheckDefaultCampaign
 4.  CheckParamAliases
 5.  ChooseStream
 6.  ChooseOffer                 ‚Äî Pick final offer URL
 7.  FindAffiliateNetwork
 8.  UpdateCosts
 9.  UpdatePayout
10.  SetCookie
11.  ExecuteAction               ‚Äî 302 redirect to affiliate offer URL
12.  CheckSendingToAnotherCampaign
13.  StoreRawClicks
```

## Data Flow

### Hot Path ‚Äî Click Processing (<5ms target)

```
1. HTTP GET /{campaign_alias}?sub_id=X&cost=Y
2. Chi router ‚Üí Traffic handler (not admin)
3. Level-1 Pipeline runs (23 stages):
   a. Build RawClick from request headers/params
   b. Lookup Campaign from Valkey cache (NOT Postgres, never on hot path)
   c. Resolve Geo: MaxMind .mmdb loaded into process memory ‚Üí sub-ms lookup
   d. Resolve Device: UA parser (mssola/device-detector)
   e. Match stream filters ‚Üí pick winning stream (weighted rotation)
   f. Rotate landing/offer (weighted round-robin, weights in Valkey)
   g. Generate click token (crypto/rand)
   h. Set visitor_code + session cookies
   i. Build 302 response (most common action type)
4. Async: push RawClick to buffered Go channel
5. Return HTTP 302 + Set-Cookie (done, <5ms)
6. Background: batch channel ‚Üí INSERT INTO ClickHouse every 500ms
```

### Landing Click Path

```
1. Visitor on landing page clicks CTA link: GET /lp/{token}/click
2. Level-2 Pipeline (13 stages):
   a. Resolve campaign from LP token (Valkey)
   b. Re-evaluate stream/offer for this visitor
   c. Execute 302 to final affiliate offer URL
3. Store landing click to ClickHouse
```

### Admin Path ‚Äî CRUD Operations (~50-200ms)

```
1. HTTP POST /api/v1/campaigns
2. Auth middleware: validate session token (Valkey lookup)
3. Controller validates POST body, writes to PostgreSQL
4. On save: invalidate + warm Valkey cache with updated entity
5. Return JSON 201
```

### Postback / Conversion Path

```
1. HTTP GET /postback/{network}/{token}?payout=X&status=lead
2. Resolve click by token (Valkey ‚Üí ClickHouse fallback)
3. Insert conversion record into ClickHouse
4. Fire configured network postback URL (fire-and-forget goroutine)
```

### Background Workers

```
1. Click Writer     ‚Äî buffered channel ‚Üí batch INSERT ClickHouse (every 500ms)
2. Session Janitor  ‚Äî expire old uniqueness sessions from Valkey
3. GeoIP Updater    ‚Äî weekly cron: replace .mmdb files + reload into memory
4. Hit Limit Reset  ‚Äî daily reset of Valkey click cap counters
```

## Database Schema Notes

**PostgreSQL** (config layer):
- Tables: campaigns, streams, stream_filters, offers, landings, domains,
  affiliate_networks, traffic_sources, users, api_keys, settings, conversions_meta
- JSONB columns for filter rules and stream action configs
- golang-migrate for schema management (embedded in binary)

**ClickHouse** (analytics layer):
- `clicks` table: partitioned by `toYYYYMM(created_at)`, ordered by `(campaign_id, created_at)`
- `conversions` table: join-able to clicks via `click_token`
- Sub-second aggregation for reporting via materialized views

**Valkey** key namespaces:
- `campaign:{id}` ‚Üí serialized Campaign struct (TTL: 1h, refreshed on admin save)
- `sess:{visitor_code}` ‚Üí uniqueness session
- `clicks:queue` ‚Üí RPUSH list for async batch writes
- `hitlimit:{stream_id}:{date}` ‚Üí click counter (INCR)
- `session:{token}` ‚Üí admin session data (TTL: 24h)

## Technical Debt (From Keitaro PHP Source ‚Äî Items We Fix)

- [ ] **MySQL for everything** ‚Äî Keitaro stores config + clicks + stats in one MySQL instance. Scalability ceiling at ~10M click rows.
- [ ] **Synchronous click writes** ‚Äî `StoreRawClicksStage` writes to DB inline, partially buffered via Redis queue but still blocks on flush.
- [ ] **ADODB legacy** ‚Äî Raw SQL string concatenation with `Db::quote()`. No parameterized queries, SQL injection risk.
- [ ] **IonCube encryption** ‚Äî Source was encrypted. Decoded quality varies; silent bugs possible.
- [ ] **Singleton pattern everywhere** ‚Äî `::instance()` on every service. Hard to test, impossible to mock.
- [ ] **No dependency injection** ‚Äî Services construct their own dependencies in constructors.
- [ ] **PHP session handling** ‚Äî Relies on PHP's session mechanism. Not distributed-safe.
- [ ] **License enforcement** ‚Äî `Component/Av/` and `Component/Branding/` enforce commercial licensing. Removed in ZAI.
- [ ] **Telemetry** ‚Äî `Component/SelfUpdate/SelfChecker/` phones home. Removed in ZAI.
- [ ] **No ClickHouse** ‚Äî Analytics are slow MySQL GROUP BY queries. We replace with columnar ClickHouse.
