## Overview

SkyPlix TDS (Traffic Distribution System) is a high-performance Go-based click tracker and traffic router, designed as a drop-in replacement for PHP-based Keitaro. It processes affiliate marketing click traffic through a multi-stage pipeline, routing visitors to the correct offer/landing page based on campaign configuration, stream filters, geo/device targeting, bot detection, and visitor binding — all targeting sub-5ms p99 latency.

The system follows a **request → pipeline → redirect** model where each incoming HTTP click is processed through an ordered sequence of stages that progressively enrich a `Payload` struct, ultimately producing an HTTP redirect (or content response) and asynchronously persisting analytics to ClickHouse.

## Data Flow

### Level 1 Click (Campaign Click)
```
1. HTTP GET /{alias}  or  GET / (bare domain)
2. DomainRedirect: resolve bare domain → campaign
3. CheckPrefetch: abort if browser prefetch
4. BuildRawClick: parse IP, UA, referrer; basic bot detection
5. FindCampaign: lookup by alias (Valkey → PG fallback)
6. CheckDefaultCampaign: fallback if alias not found
7. UpdateRawClick: enrich with GeoIP + device detection
8. CheckParamAliases: map traffic source parameters
9. UpdateCampaignUniqueness: check/set campaign-level unique visitor
10. ChooseStream: 3-tier selection (FORCED→REGULAR→DEFAULT) with filters + limits
11. UpdateStreamUniqueness: check/set stream-level unique visitor
12. ChooseLanding: weighted rotation + visitor binding
13. ChooseOffer: weighted rotation + visitor binding
14. SaveLPToken: persist L1→L2 link token
15. GenerateToken: 32-char hex click token
16. FindAffiliateNetwork: resolve network for postback
17. UpdateHitLimit: increment daily/total counters
18. UpdateCosts: apply cost model
19. UpdatePayout: apply offer payout
20. SaveUniquenessSession: persist uniqueness to Valkey
21. SetCookie: set visitor binding cookie
22. ExecuteAction: send HTTP response (302/content/proxy)
23. StoreRawClicks: push to ClickHouse batch channel (AlwaysRun)
```

### Level 2 Click (Landing → Offer)
```
1. BuildRawClick: parse request
2. L2FindCampaign: resolve campaign/stream via LP token
3. ChooseOffer: select offer (weighted rotation + binding)
4. FindAffiliateNetwork: resolve network
5. UpdateCosts/UpdatePayout: financial attribution
6. GenerateToken: new click token
7. SetCookie: visitor cookie
8. ExecuteAction: 302 redirect to offer URL
9. StoreRawClicks: persist to ClickHouse
```

## Conventions

### Naming
- **Packages:** Lowercase single-word (`cache`, `filter`, `action`, `pipeline`)
- **Files:** Descriptive with stage numbers (`9_choose_stream.go`)
- **Types:** PascalCase, domain-driven (`RawClick`, `StreamFilter`, `CacheWarmupWorker`)
- **Interfaces:** Single-method or small (`Stage`, `Worker`, `Filter`, `Action`, `Weightable`)

### Structure
- **`cmd/`** — One entry point (`zai-tds`)
- **`internal/`** — 22 packages, each focused on one concern
- **`db/`** — SQL migrations separated by database (postgres/clickhouse)
- **Per-package documentation:** Every package has an `AGENTS.md` file

### Error Handling
- Pipeline stage errors abort with wrapped fmt.Errorf
- Cache misses fall through to PostgreSQL (no errors to caller)
- ClickHouse write failures are logged but don't crash
- UUID parse failures use nil UUID fallback (prevents batch failures)

### Testing
- Build-tagged integration tests (`//go:build integration`)
- Test suites in `test/integration/`: `click_test.go`, `routing_test.go`, `admin_test.go`
- Testdata fixtures in `test/integration/testdata/`
- Unit test stubs in `test/unit/{queue,worker}`

### Configuration
- YAML primary (`config.yaml`) with env var overrides
- Validation with debug-mode exceptions (salt requirement)
- Docker Compose for local development
